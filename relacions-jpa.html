Relacions JPA

Com mapem les relacions entre entitats a la base de dades — @OneToOne, @OneToMany, @ManyToMany — i com controlem el comportament amb cascade, fetch, mappedBy i @JoinColumn.

OneToOne OneToMany ManyToMany Cascade Fetch JPA Spring Boot @JsonIgnore

1. El model del projecte

El projecte gestiona una escola amb quatre entitats relacionades entre elles:

Student ──────────────── AcademicRecord @OneToOne (bidireccional) │ ├─────────────── Absence (*) @OneToMany / @ManyToOne │ └─────────────── Course (*) @ManyToMany └──────── Student (*) (taula pont)

Cada relació té un costat propietari (qui guarda la clau forana a la BD) i un costat invers (qui usa mappedBy). Entendre aquesta distinció és clau per saber quines columnes es creen a la base de dades.

Codi complet de les entitats

model/Student.java MODEL 
@Entity
@Table(name = "students")
public class Student {

    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;
    private String firstName;
    private String lastName;
    private String email;

    // Costat INVERS de la relació OneToOne
    @OneToOne(mappedBy = "student", cascade = CascadeType.ALL, fetch = FetchType.LAZY)
    private AcademicRecord academicRecord;

    // Costat INVERS de la relació OneToMany
    @OneToMany(mappedBy = "student", cascade = CascadeType.ALL)
    private List<Absence> absences = new ArrayList<>();

    // Costat PROPIETARI de la relació ManyToMany
    @ManyToMany
    @JoinTable(
        name = "student_course_rel",
        joinColumns        = @JoinColumn(name = "student_id"),
        inverseJoinColumns = @JoinColumn(name = "course_id")
    )
    private List<Course> courses = new ArrayList<>();

    // Constructors
    public Student() {}
    public Student(String firstName, String lastName, String email) {
        this.firstName = firstName;
        this.lastName  = lastName;
        this.email     = email;
    }

    // Getters i Setters
    public Long   getId()                    { return id; }
    public void   setId(Long id)               { this.id = id; }

    public String getFirstName()               { return firstName; }
    public void   setFirstName(String firstName) { this.firstName = firstName; }

    public String getLastName()                { return lastName; }
    public void   setLastName(String lastName)  { this.lastName = lastName; }

    public String getEmail()                   { return email; }
    public void   setEmail(String email)        { this.email = email; }

    public AcademicRecord getAcademicRecord() { return academicRecord; }
    // Setter especial: manté els dos costats sincronitzats
    public void setAcademicRecord(AcademicRecord academicRecord) {
        this.academicRecord = academicRecord;
        if (academicRecord != null) {
            academicRecord.setStudent(this);
        }
    }

    public List<Absence> getAbsences()                   { return absences; }
    public void            setAbsences(List<Absence> absences) { this.absences = absences; }

    public List<Course> getCourses()                  { return courses; }
    public void           setCourses(List<Course> courses) { this.courses = courses; }

    // Mètodes helper per a la relació OneToMany
    public void addAbsence(Absence absence) {
        absences.add(absence);
        absence.setStudent(this);
    }
    public void removeAbsence(Absence absence) {
        absences.remove(absence);
        absence.setStudent(null);
    }
}

2. Propietari vs costat invers

En tota relació bidireccional JPA hi ha exactament un costat propietari i un costat invers. Entendre això és fonamental per saber com JPA escriu a la base de dades.

L'analogia per entendre-ho

Imagina dos amics, Anna i Expedient. La targeta d'identificació de l'Expedient porta escrit el DNI d'Anna. Si vols saber a qui pertany l'expedient, mires la targeta de l'Expedient — ell és el propietari de la relació. Anna sap que té un expedient, però no porta res escrit: és el costat invers.

A la base de dades passa el mateix: qui té la clau forana a la seva taula és el propietari.

Regla d'or per identificar el propietari

Busca qui té @JoinColumn o @JoinTable: aquell és el propietari. L'altre costat sempre tindrà mappedBy.
En una relació @ManyToOne / @OneToMany, el costat @ManyToOne és sempre el propietari.

Característica Costat propietari Costat invers
Anotació @JoinColumn o @JoinTable mappedBy = "..."
Crea columna FK a la BD? Sí — és qui guarda l'ID de l'altre No — només llegeix la relació
JPA escolta canvis? Sí — si el modifiques, JPA l'escriu a la BD No — si el modifiques, JPA ho ignora
Exemple al projecte AcademicRecord.student, Absence.student Student.academicRecord, Student.absences

Què significa el valor de mappedBy?

El string que poses a mappedBy ha de ser exactament el nom del camp Java de l'altra classe que gestiona la relació (el propietari). No és el nom de la columna a la BD, és el nom de l'atribut Java.

Exemple — com llegir mappedBy MODEL 
// A Absence.java — costat PROPIETARI
@ManyToOne
@JoinColumn(name = "student_id")
private Student student;   // ← el camp es diu "student"
                              //   (nom del camp Java, no de la columna BD)

// A Student.java — costat INVERS
@OneToMany(mappedBy = "student")  // ← ha de coincidir amb el nom del camp de dalt
private List<Absence> absences;
Error freqüent

Si el valor de mappedBy no coincideix exactament amb el nom del camp del costat propietari, JPA llançarà una excepció en arrencar l'aplicació. Si oblides posar mappedBy al costat invers, JPA crearà una columna extra innecessària a la BD.

3. @OneToOne — Student ↔ AcademicRecord

Un estudiant té exactament un expedient acadèmic, i un expedient pertany a exactament un estudiant.

Costat propietari — AcademicRecord

AcademicRecord és el propietari perquè té @JoinColumn. Això significa que la taula records tindrà una columna student_id.

model/AcademicRecord.java MODEL 
@OneToOne(fetch = FetchType.LAZY)
@JoinColumn(name = "student_id", unique = true)
private Student student;
  • @JoinColumn(name = "student_id") → crea la columna student_id a la taula records.
  • unique = true → garanteix que dos expedients no apuntin al mateix estudiant.
  • fetch = FetchType.LAZY → no carrega l'estudiant fins que es cridi getStudent().

Costat invers — Student

Student és el costat invers: usa mappedBy i no crea cap columna nova.

model/Student.java MODEL 
@OneToOne(mappedBy = "student", cascade = CascadeType.ALL, fetch = FetchType.LAZY)
private AcademicRecord academicRecord;
  • mappedBy = "student" → indica que la relació ja està mapada pel camp student de AcademicRecord.
  • cascade = CascadeType.ALL → si esborres o guardes un Student, l'operació es propaga a l'AcademicRecord.

Per qué cal el setter especial? — El problema sense ell

Amb una relació bidireccional, JPA només mira el costat propietari per saber si ha d'escriure la clau forana a la BD. Si actualitzes només el costat invers, JPA no escriurà res:

❌ Codi INCORRECTE — la FK no es guarda SERVICE 
Student student = new Student("Anna", "Garcia", "anna@escola.cat");
AcademicRecord record = new AcademicRecord("EXP-001", "ACTIVE");

// Actualitzem NOMÉS el costat invers (Student)
student.setAcademicRecord(record);   // sense el setter especial

studentRepository.save(student);

// Resultat a la BD:
// records → student_id = NULL  ← JPA ignora el costat invers!
✓ Codi CORRECTE — amb el setter especial SERVICE 
Student student = new Student("Anna", "Garcia", "anna@escola.cat");
AcademicRecord record = new AcademicRecord("EXP-001", "ACTIVE");

// El setter especial actualitza els DOS costats
student.setAcademicRecord(record);   // aquest setter ja crida record.setStudent(student)

studentRepository.save(student);

// Resultat a la BD:
// records → student_id = 1  ← correcte ✓

El setter especial a l'entitat

Amb relacions bidireccionals cal actualitzar els dos costats manualment. El setter especial de Student ho fa automàticament en una sola crida:

model/Student.java — setAcademicRecord MODEL 
public void setAcademicRecord(AcademicRecord academicRecord) {
    this.academicRecord = academicRecord;       // 1) actualitza el costat invers (Student)
    if (academicRecord != null) {
        academicRecord.setStudent(this);       // 2) actualitza el costat PROPIETARI (AcademicRecord)
    }                                            //    → JPA ara sí escriurà student_id a la BD
}

Ús complet al servei

service/StudentService.java — crear estudiant amb expedient SERVICE 
public Student createStudentWithRecord(String firstName, String lastName,
                                          String email, String recordNumber) {
    // 1. Crear les dues entitats
    Student student = new Student(firstName, lastName, email);
    AcademicRecord record = new AcademicRecord(recordNumber, "ACTIVE");

    // 2. Vincular-les amb el setter especial (sincronitza els dos costats)
    student.setAcademicRecord(record);

    // 3. Guardar l'estudiant — cascade = ALL guarda l'AcademicRecord automàticament
    //    JPA genera: INSERT INTO students ... i INSERT INTO records (student_id=?) ...
    return studentRepository.save(student);
}

Taules resultants a la BD

students records ────────────────── ────────────────────────────── id (PK) id (PK) first_name record_number last_name status email student_id (FK → students.id, UNIQUE)

4. @OneToMany / @ManyToOne — Student ↔ Absence

Un estudiant pot tenir moltes absències, però cada absència pertany a un sol estudiant.

Costat propietari — Absence (@ManyToOne)

model/Absence.java MODEL 
@ManyToOne
@JoinColumn(name = "student_id")
private Student student;
  • El costat @ManyToOne és sempre el propietari.
  • @JoinColumn(name = "student_id") → crea la columna student_id a la taula abcences.

Costat invers — Student (@OneToMany)

model/Student.java MODEL 
@OneToMany(mappedBy = "student", cascade = CascadeType.ALL)
private List<Absence> absences = new ArrayList<>();

Per qué cal el mètode helper? — El problema sense ell

La mateixa regla que a l'@OneToOne: JPA només escolta el costat propietari (Absence). Si modifiques la llista de Student sense actualitzar Absence.student, la clau forana quedarà a NULL:

❌ Codi INCORRECTE — la FK queda NULL SERVICE 
Absence absence = new Absence(LocalDate.now(), false);

// Actualitzem NOMÉS la llista (costat invers)
student.getAbsences().add(absence);

studentRepository.save(student);

// Resultat a la BD:
// abcences → student_id = NULL  ← JPA ignora el costat invers!
✓ Codi CORRECTE — amb el mètode helper SERVICE 
Absence absence = new Absence(LocalDate.now(), false);

// addAbsence sincronitza els dos costats internament
student.addAbsence(absence);

studentRepository.save(student);

// Resultat a la BD:
// abcences → student_id = 1  ← correcte ✓

Els mètodes helper a l'entitat

Com que JPA només escolta canvis al costat propietari (Absence), els mètodes helper s'asseguren que els dos costats estiguin sincronitzats:

model/Student.java — addAbsence / removeAbsence MODEL 
public void addAbsence(Absence absence) {
    absences.add(absence);        // 1) actualitza la llista del costat invers (Student)
    absence.setStudent(this);   // 2) actualitza el camp del costat PROPIETARI (Absence)
}                                //    → JPA ara sí escriurà student_id a la BD

public void removeAbsence(Absence absence) {
    absences.remove(absence);
    absence.setStudent(null);   // posa student_id = NULL → l'absència queda desvinculada
}

Ús complet al servei

service/AbsenceService.java — afegir absència a un estudiant SERVICE 
public Absence addAbsenceToStudent(Long studentId, LocalDate date, boolean justified) {

    // 1. Recuperar l'estudiant de la BD
    Student student = studentRepository.findById(studentId)
        .orElseThrow(() -> new RuntimeException("Estudiant no trobat"));

    // 2. Crear la nova absència
    Absence absence = new Absence(date, justified);

    // 3. Vincular: addAbsence sincronitza els dos costats
    student.addAbsence(absence);

    // 4. Guardar l'estudiant — cascade = ALL guarda l'Absence automàticament
    //    JPA genera: INSERT INTO abcences (date, justified, student_id) VALUES (?, ?, ?)
    studentRepository.save(student);

    return absence;
}
Compte

Si afegeixes una Absence només a la llista de Student però no assignes absence.setStudent(this), JPA no guardarà la clau forana correctament perquè el propietari és Absence. Usa sempre el mètode addAbsence() en lloc de getAbsences().add().

Taules resultants a la BD

students abcences ────────────── ───────────────────────────── id (PK) id (PK) first_name date last_name justified email student_id (FK → students.id)

5. @ManyToMany — Student ↔ Course

Un estudiant pot estar matriculat a molts cursos, i un curs pot tenir molts estudiants. JPA necessita una taula pont per representar-ho.

Costat propietari — Student

El costat propietari és qui declara @JoinTable, que configura el nom i les columnes de la taula pont.

model/Student.java MODEL 
@ManyToMany
@JoinTable(
    name = "student_course_rel",          // nom de la taula pont
    joinColumns        = @JoinColumn(name = "student_id"),  // columna del propietari
    inverseJoinColumns = @JoinColumn(name = "course_id")    // columna de l'altre costat
)
private List<Course> courses = new ArrayList<>();

Costat invers — Course

model/Course.java MODEL 
@ManyToMany(mappedBy = "courses")
private List<Student> students = new ArrayList<>();

Taules resultants a la BD

students student_course_rel course ────────── ─────────────────────── ────────────── id (PK) <─── student_id (FK) id (PK) ... course_id (FK) ───> name level
@JoinTable — paràmetres

name → nom de la taula pont.
joinColumns → columna que apunta a l'entitat propietària.
inverseJoinColumns → columna que apunta a l'altra entitat.

6. CascadeType — propagació d'operacions

cascade defineix si les operacions sobre l'entitat pare es propaguen automàticament als fills.

TipusEfecteQuan usar-lo
ALL Propaga totes les operacions (persist, merge, remove, refresh, detach) Quan el fill no pot existir sense el pare
PERSIST Guarda els fills quan guardes el pare Creació en cascada sense voler esborrar en cascada
REMOVE Esborra els fills quan esborres el pare Eliminació en cascada d'entitats dependents
MERGE Actualitza els fills quan actualitzes el pare Sincronització d'actualitzacions

Cascade al projecte

model/Student.java — cascade examples MODEL 
// Esborrar Student esborra l'AcademicRecord
@OneToOne(mappedBy = "student", cascade = CascadeType.ALL, ...)
private AcademicRecord academicRecord;

// Esborrar Student esborra totes les seves Absence
@OneToMany(mappedBy = "student", cascade = CascadeType.ALL)
private List<Absence> absences;

// Course NO té cascade — esborrar Student no esborra els cursos
@ManyToMany
private List<Course> courses;
Per què Course no té cascade?

Els cursos poden tenir molts estudiants. Si un estudiant s'esborra i el cascade esborres el curs, tots els altres estudiants perdrien el curs. Sense cascade, esborrar un estudiant només elimina la seva fila a la taula pont student_course_rel, no el curs en si.

Pas a pas — què passa quan cridem save() i delete()

Amb cascade = ALL configurat, JPA fa automàticament el treball que hauríem de fer manualment:

Escenari 1 — guardar un estudiant nou amb expedient i absències SERVICE 
Student student = new Student("Anna", "Garcia", "anna@escola.cat");
AcademicRecord record = new AcademicRecord("EXP-001", "ACTIVE");
Absence abs1 = new Absence(LocalDate.of(2024, 3, 10), false);

student.setAcademicRecord(record);   // vincular expedient
student.addAbsence(abs1);            // vincular absència

studentRepository.save(student);
// JPA executa (per ordre):
// 1. INSERT INTO students  (first_name, ...) VALUES ('Anna', ...)  → id=1
// 2. INSERT INTO records   (record_number, status, student_id) VALUES ('EXP-001','ACTIVE',1)
// 3. INSERT INTO abcences  (date, justified, student_id) VALUES ('2024-03-10', false, 1)
Escenari 2 — esborrar l'estudiant SERVICE 
studentRepository.deleteById(1L);
// cascade = ALL propaga el DELETE:
// 1. DELETE FROM abcences  WHERE student_id = 1  ← cascade REMOVE
// 2. DELETE FROM records   WHERE student_id = 1  ← cascade REMOVE
// 3. DELETE FROM students  WHERE id = 1
// (la fila de student_course_rel s'esborra sola gràcies a la FK constraint)
Escenari 3 — sense cascade (comportament per defecte) SERVICE 
// Si NO hi hagués cascade, hauríem de fer-ho manualment:
absenceRepository.deleteAllByStudentId(1L);
academicRecordRepository.deleteByStudentId(1L);
studentRepository.deleteById(1L);
// Amb cascade = ALL, tot això es fa en una sola crida ↑

7. FetchType — quan es carreguen les dades

Controla quan JPA executa el SELECT a la BD per carregar les entitats relacionades.

TipusQuan carrega?Per defecte a...
LAZY Només quan s'accedeix al camp per primera vegada @OneToMany, @ManyToMany
EAGER Immediatament, en la mateixa consulta del pare @OneToOne, @ManyToOne
model/AcademicRecord.java — LAZY fetch MODEL 
@OneToOne(fetch = FetchType.LAZY)
@JoinColumn(name = "student_id", unique = true)
private Student student;

// Quan carregues un AcademicRecord:
// SELECT * FROM records WHERE id = ?   → OK, carrega l'expedient
// academicRecord.getStudent()          → ara fa: SELECT * FROM students WHERE id = ?

LAZY vs EAGER — quin triar?

  • LAZY (recomanat per defecte): evita carregar dades que potser no necessites. Millora el rendiment quan hi ha moltes relacions.
  • EAGER: útil quan sempre necessis la relació. Però si en tens moltes, pot generar consultes SQL massives (problema N+1).
Problema N+1

Si tens 100 estudiants amb EAGER i cada un té cursos, JPA farà 1 SELECT per obtenir els estudiants i 100 SELECTs addicionals per obtenir els cursos de cadascun. Amb LAZY + consultes optimitzades (JOIN FETCH) s'evita aquest problema.

8. El problema de les referències circulars

En una API REST, quan Spring intenta serialitzar una entitat JPA a JSON amb relacions bidireccionals, pot entrar en un bucle infinit:

GET /api/students/1 → Student │ serialitza academicRecord │ AcademicRecord │ serialitza student │ Student ←──── de nou el mateix objecte! │ serialitza academicRecord ... ∞ BUCLE INFINIT

Solució del projecte: DTOs i Mappers

En lloc de retornar l'entitat JPA directament, el Mapper construeix un DTO sense incloure la referència de tornada:

mapper/StudentMapper.java — toDto (tall del cicle) MAPPER 
public StudentDTO toDto(Student student) {
    if (student == null) return null;

    StudentDTO dto = new StudentDTO(
        student.getId(),
        student.getFirstName(),
        student.getLastName(),
        student.getEmail()
    );

    if (student.getAcademicRecord() != null) {
        AcademicRecordDTO record = new AcademicRecordDTO(
            student.getAcademicRecord().getId(),
            student.getAcademicRecord().getRecordNumber(),
            student.getAcademicRecord().getStatus()
            // NO s'afegeix "student" → cicle tallat ✓
        );
        dto.setAcademicRecord(record);
    }

    return dto;
}

JSON resultant — sense cicle

Resposta GET /api/students/1 JSON 
{
  "id": 1,
  "firstName": "Anna",
  "lastName": "Garcia",
  "email": "anna@escola.cat",
  "academicRecord": {
    "id": 1,
    "recordNumber": "EXP-2024-001",
    "status": "ACTIVE"
    // cap camp "student" aquí → cicle tallat ✓
  }
}
Altres solucions possibles (menys recomanades)

@JsonIgnore — ignora el camp en la serialització JSON.
@JsonManagedReference / @JsonBackReference — anota els dos costats de la relació.
@JsonIdentityInfo — substitueix objectes repetits per un identificador.

9. @JsonIgnore — controlar la serialització JSON

@JsonIgnore és una anotació de la llibreria Jackson (el serialitzador JSON que usa Spring Boot per defecte) que indica al serialitzador que ometi un camp quan converteix un objecte Java a JSON — i, per defecte, també en la deserialització (JSON → Java).

Dependència

@JsonIgnore forma part de com.fasterxml.jackson.annotation.JsonIgnore. Spring Boot ja inclou Jackson automàticament via spring-boot-starter-web, no cal cap dependència addicional.

Sintaxi bàsica

Exemple — camp ignorat MODEL 
import com.fasterxml.jackson.annotation.JsonIgnore;

@Entity
public class Absence {

    private Long   id;
    private LocalDate date;

    @JsonIgnore          // aquest camp NO apareixerà al JSON de sortida
    private Student student;
}

Per a què serveix — els tres casos principals

1. Trencar cicles de referències circulars (cas més freqüent)

En relacions bidireccionals JPA, si retornes l'entitat directament a la API, Jackson entra en un bucle infinit: Student → absences → Absence → student → Student → ...

Posant @JsonIgnore al costat invers o al costat propietari (el que no necessites a la resposta), talles el cicle:

model/Absence.java — @JsonIgnore al @ManyToOne MODEL 
@ManyToOne
@JoinColumn(name = "student_id")
@JsonIgnore           // quan serialitzem Absence, no incloem l'objecte Student
private Student student;

Resultat JSON quan obtens les absències d'un estudiant:

Resposta GET /api/students/1/absences JSON 
[
  {
    "id": 1,
    "date": "2024-03-15",
    "justified": false
    // "student" no apareix → cicle tallat ✓
  }
]

2. Ocultar dades sensibles

Si una entitat té camps confidencials (contrasenya, token, dades internes), @JsonIgnore impedeix que surtin a la resposta JSON:

model/User.java — ocultar camp sensible MODEL 
public class User {
    private Long   id;
    private String username;

    @JsonIgnore
    private String password;  // mai ha de sortir al JSON de resposta
}

3. Evitar camps redundants o pesats

Camps calculats, col·leccions molt grans o dades que mai necessitarà el client es poden ignorar per reduir la mida de la resposta.

@JsonIgnore a les relacions del projecte

Aplicat al projecte school, es posaria al costat de la relació que no ha de ser el punt d'entrada a la serialització, evitant el cicle sense necessitat de DTOs:

model/AcademicRecord.java — OneToOne sense cicle MODEL 
@OneToOne(fetch = FetchType.LAZY)
@JoinColumn(name = "student_id", unique = true)
@JsonIgnore   // AcademicRecord no serialitza l'objecte Student complet
private Student student;
model/Absence.java — ManyToOne sense cicle MODEL 
@ManyToOne
@JoinColumn(name = "student_id")
@JsonIgnore   // Absence no serialitza l'objecte Student complet
private Student student;

@JsonIgnoreProperties — variant a nivell de classe

En lloc d'anotar cada camp, pots especificar a nivell de classe quins camps ignorar. Útil quan treballes amb classes que no pots modificar directament o quan vols ignorar múltiples camps alhora:

Exemple — @JsonIgnoreProperties MODEL 
@JsonIgnoreProperties({"student", "internalNotes"})
@Entity
public class AcademicRecord {
    // els camps "student" i "internalNotes" seran ignorats en tota la classe
    ...
}

@JsonIgnore vs DTO — quan usar cadascun

Criteri @JsonIgnore DTO + Mapper
Complexitat d'implementació Molt baixa — una anotació Alta — cal crear DTOs i Mappers
Flexibilitat per endpoint Cap — s'aplica a tota la classe Total — cada endpoint pot tenir el seu DTO
Control de la resposta Limitat — amaga camps, no en transforma Total — pots transformar i combinar camps
Separació de capes Pobra — l'entitat JPA sap de JSON Bona — cada capa té la seva responsabilitat
Validació d'entrada Compartida amb l'entitat Separada al DTO d'entrada
Recomanat per a... Prototips, APIs senzilles, camps sensibles APIs de producció, lògica de negoci complexa
Limitació clau de @JsonIgnore

@JsonIgnore afecta tots els endpoints alhora. Si un endpoint necessita el camp i un altre no, hauràs de crear DTOs igualment. Per exemple, si GET /absences no necessita l'estudiant però GET /absences/{id}/detail sí, @JsonIgnore no és suficient.

@JsonIgnore i deserialització

Per defecte, @JsonIgnore ignora el camp tant en lectura (serialització) com en escriptura (deserialització). Si vols ignorar-lo només en la sortida però acceptar-lo en l'entrada, usa: @JsonProperty(access = JsonProperty.Access.READ_ONLY).
