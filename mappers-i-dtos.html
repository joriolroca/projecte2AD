DTOs i Mappers

Com convertim entitats JPA en respostes d'API — i peticions HTTP en objectes persistibles — sense barrejar capes ni exposar l'esquema de base de dades.

DTO Mapper Service Controller Spring Boot

1. Què són els DTOs?

Un DTO (Data Transfer Object) és una classe Java senzilla — sense lògica de negoci — el propòsit de la qual és transportar dades entre capes. En una API REST distingim dos tipus:

Tipus Descripció Exemples al projecte
Request DTO Representa el cos de la petició HTTP entrant. Conté els camps que el client pot enviar. StudentCreateRequest, AcademicRecordRequest
Response DTO Representa la resposta que l'API retorna al client. S'usa per controlar quina informació s'exposa. StudentDTO, AcademicRecordDTO

Per què no retornem l'entitat JPA directament?

Les entitats JPA (@Entity) estan acoblades a la base de dades: contenen anotacions de persistència, relacions bidireccionals i, sovint, camps que no volem exposar (contrasenyes, claus foranes internes, etc.). Els DTOs permeten desacoblar el que es guarda del que s'envia o es rep.

2. Què són els Mappers?

Un Mapper és una classe de Spring (@Component) que s'encarrega de la conversió entre entitats i DTOs. Centralitza tota la lògica de transformació en un sol lloc, de manera que ni el servei ni el controlador han de saber com es construeix un DTO.

Convenció de noms

Per convenció, els mètodes d'un mapper es diuen toDto() i toEntity(). Alguns projectes usen toDTO() (majúscules) — al repositori, AcademicRecordMapper usa toDTO() mentre que StudentMapper usa toDto(). Ambdues formes són correctes, però és recomanable ser consistent dins del projecte.

Mètode Entrada Sortida Quins en fa servir
toDto() / toDTO() Entitat JPA Response DTO Servei (operacions de lectura i creació)
toEntity() Request DTO Entitat JPA Servei (creació i actualització)

4. DTOs del projecte

StudentDTO — Response

S'usa per retornar informació d'un estudiant al client. Inclou un camp academicRecord de tipus AcademicRecordDTO que permet retornar l'expedient acadèmic aniuat dins la mateixa resposta.

dto/StudentDTO.java Response DTO 
public class StudentDTO {
    private Long id;
    private String firstName;
    private String lastName;
    private String email;
    private AcademicRecordDTO academicRecord; // DTO aniuat, no entitat JPA

    // Constructor sense academicRecord (per evitar recursivitat)
    public StudentDTO(Long id, String firstName, String lastName, String email) { ... }

    // Getters i Setters per tots els camps
    public AcademicRecordDTO getAcademicRecord() { return academicRecord; }
    public void setAcademicRecord(AcademicRecordDTO academicRecord) { this.academicRecord = academicRecord; }
}

Camp opcional — s'assigna després de la construcció

El camp academicRecord no s'inclou al constructor principal. El Mapper el comprova per null i el seta amb un setter quan existeix. Això evita errors quan un estudiant no té expedient assignat.

5. Mappers del projecte

StudentMapper

Gestiona la conversió bidireccional per a l'entitat Student. Té dos mètodes: toDto() per les lectures i toEntity() per les creacions.

mapper/StudentMapper.java Mapper 
@Component   // Spring el registra com a bean injectable
public class StudentMapper {

    // ── Entitat JPA → Response DTO ─────────────────────────────────────
    public StudentDTO toDto(Student student) {
        if (student == null) return null;  // Null-safe

        StudentDTO dto = new StudentDTO(
            student.getId(),
            student.getFirstName(),
            student.getLastName(),
            student.getEmail()
        );

        // Si l'estudiant té expedient, el convertim a DTO aniuat
        if (student.getAcademicRecord() != null) {
            AcademicRecordDTO record = new AcademicRecordDTO(
                student.getAcademicRecord().getId(),
                student.getAcademicRecord().getRecordNumber(),
                student.getAcademicRecord().getStatus()
                // NOTA: no afegim student de nou → evitem referència circular
            );
            dto.setAcademicRecord(record);
        }

        return dto;
    }

    // ── Request DTO → Entitat JPA ──────────────────────────────────────
    public Student toEntity(StudentCreateRequest request) {
        if (request == null) return null;  // Null-safe

        Student student = new Student(
            request.getFirstName(),
            request.getLastName(),
            request.getEmail()
        );

        // Creem l'expedient acadèmic si el client n'ha enviat el número
        if (request.getRecordNumber() != null) {
            AcademicRecord record = new AcademicRecord(
                request.getRecordNumber(),
                request.getRecordStatus() != null
                    ? request.getRecordStatus()
                    : "ACTIVE"  // Valor per defecte
            );
            student.setAcademicRecord(record);
        }

        return student;
    }
}

Punts clau de StudentMapper

  • Null-safety: tots dos mètodes comproven null a l'inici i retornen null si l'entrada és nul·la.
  • Creació en cascada: toEntity() pot crear un AcademicRecord associat si el request ho inclou — gràcies al camp recordNumber.
  • Evitar recursivitat: quan toDto() crea l'AcademicRecordDTO aniuat, usa el constructor de 3 paràmetres que no inclou l'estudiant, trencant el cicle.

AcademicRecordMapper

Gestiona la conversió per a AcademicRecord. Quan retorna el DTO, inclou l'estudiant associat com a StudentDTO, però sense l'expedient d'aquest estudiant.

mapper/AcademicRecordMapper.java Mapper 
@Component
public class AcademicRecordMapper {

    // ── Entitat JPA → Response DTO ─────────────────────────────────────
    public AcademicRecordDTO toDTO(AcademicRecord entity) {
        if (entity == null) return null;

        AcademicRecordDTO dto = new AcademicRecordDTO(
            entity.getId(),
            entity.getRecordNumber(),
            entity.getStatus()
        );

        // Afegim l'estudiant associat si existeix
        if (entity.getStudent() != null) {
            StudentDTO student = new StudentDTO(
                entity.getStudent().getId(),
                entity.getStudent().getFirstName(),
                entity.getStudent().getLastName(),
                entity.getStudent().getEmail()
                // NOTA: no afegim academicRecord → trenca la recursivitat
            );
            dto.setStudent(student);
        }

        return dto;
    }

    // ── Request DTO → Entitat JPA ──────────────────────────────────────
    public AcademicRecord toEntity(AcademicRecordRequest request) {
        AcademicRecord entity = new AcademicRecord();
        entity.setRecordNumber(request.getRecordNumber());
        entity.setStatus(request.getStatus());
        // NOTA: no assignem l'estudiant aquí — ho fa el servei
        return entity;
    }
}

Referència circular — com s'evita

Student té un AcademicRecord i AcademicRecord té un Student. Si els dos DTOs s'inclouen mútuament sense restriccions, el serialitzador JSON entraria en un bucle infinit. Els mappers ho resolen construint l'objecte aniuat manualment sense tornar a incloure la referència de retorn:

Quan el Mapper genera un StudentDTO: StudentDTO { id, firstName, lastName, email, academicRecord: AcademicRecordDTO { id, recordNumber, status // ← NO s'inclou "student" aquí → cicle tallat ✓ } } Quan el Mapper genera un AcademicRecordDTO: AcademicRecordDTO { id, recordNumber, status, student: StudentDTO { id, firstName, lastName, email // ← NO s'inclou "academicRecord" aquí → cicle tallat ✓ } }

6. Com afecta el Mapper al Servei

Regla general del Servei

El servei mai construeix DTOs ni entitats manualment. Delega completament al Mapper. Només s'ocupa de la lògica de negoci (validacions, crides al repositori, orquestració).

StudentService — injecció i ús del Mapper

StudentService injecta el StudentMapper per constructor (injectable final — patró recomanat a Spring):

@Service
public class StudentService {

    private final StudentRepository studentRepository;
    private final StudentMapper studentMapper;  // Mapper injectat per constructor

    public StudentService(StudentRepository studentRepository,
                           StudentMapper studentMapper) {
        this.studentRepository = studentRepository;
        this.studentMapper = studentMapper;
    }
    ...
}

Lectura per ID — mapper a la sortida

service/StudentService.java — getStudentById Service 
public StudentDTO getStudentById(Long id) {
    Optional<Student> student = studentRepository.findById(id);  // 1. Crida al repo → entitat
    if (student.isPresent()) {
                return studentMapper.toDto(student.get());  // 2. Mapper: entitat → DTO
    }
    return null;
}

Llistar tots — mapper en bucle

service/StudentService.java — getAllStudents Service 
public List<StudentDTO> getAllStudents() {
    List<Student> students = studentRepository.findAll();  // 1. Repo → llista d'entitats

    List<StudentDTO> studentsListDTO = new ArrayList<>();
    for (Student item : students) {
                StudentDTO dto = studentMapper.toDto(item);  // 2. Mapper: cada entitat → DTO
        studentsListDTO.add(dto);
    }
    return studentsListDTO;
}

Crear — mapper a l'entrada i a la sortida

service/StudentService.java — createStudent Service 
public StudentDTO createStudent(StudentCreateRequest request) {
    // Validació de negoci (el repositori no en fa)
    if (studentRepository.existsByEmail(request.getEmail())) {
        throw new Error("L'email ja existeix: " + request.getEmail());
    }

        Student student = studentMapper.toEntity(request);         // 1. Mapper: Request → Entitat
    Student savedStudent = studentRepository.save(student);       // 2. Repo: guarda l'entitat
        return studentMapper.toDto(savedStudent);                   // 3. Mapper: Entitat → DTO
}

Dos usos del mapper a createStudent

Observa que a createStudent el mapper s'usa dues vegades: primer per convertir el request en entitat (entrada), i després per convertir l'entitat guardada en DTO (sortida). Sempre es retorna el DTO obtingut de la BD — no el request original — perquè la BD pot afegir valors com l'ID autogenerat.

AcademicRecordService — injecció per @Autowired

AcademicRecordService usa la injecció per camps amb @Autowired (alternativa a la injecció per constructor).

service/AcademicRecordService.java Service 
@Service
public class AcademicRecordService {

    @Autowired
    private AcademicRecordRepository recordRepository;

    @Autowired
    private AcademicRecordMapper mapper;  // Mapper injectat per @Autowired

    // ── Consultar tots ────────────────────────────────────────────────
    public List<AcademicRecordDTO> findAll() {
        List<AcademicRecord> entities = recordRepository.findAll();  // Repo → entitats

        List<AcademicRecordDTO> dtos = new ArrayList<>();
        for (AcademicRecord record : entities) {
                        AcademicRecordDTO dto = mapper.toDTO(record);  // Mapper: entitat → DTO
            dtos.add(dto);
        }
        return dtos;
    }

    // ── Consultar per ID ──────────────────────────────────────────────
    public AcademicRecordDTO findById(Long id) {
        Optional<AcademicRecord> record = recordRepository.findById(id);
        if (record.isPresent()) {
                        return mapper.toDTO(record.get());  // Mapper: entitat → DTO
        }
        return null;
    }

    // ── Modificar ─────────────────────────────────────────────────────
    public AcademicRecordDTO update(Long id, AcademicRecordRequest request) {
                AcademicRecord academic = mapper.toEntity(request);         // Mapper: Request → Entitat
        AcademicRecord academicUpdated = recordRepository.save(academic); // Repo: guarda
                return mapper.toDTO(academicUpdated);                      // Mapper: Entitat → DTO
    }
}

7. Com afecta el Mapper al Controlador

El controlador no usa mai el Mapper directament

El controlador mai crida el mapper. Treballa exclusivament amb DTOs: rep Request DTOs del cos HTTP i retorna Response DTOs al client. L'impacte del mapper al controlador és indirecte: el controlador ha de declarar els tipus DTO correctes als seus mètodes.

StudentController

controller/StudentController.java Controller 
@RestController
@RequestMapping("/api/students")
public class StudentController {

    private final StudentService studentService;  // Injecció per constructor

    public StudentController(StudentService studentService) {
        this.studentService = studentService;
    }

    // ── POST /api/students ─────────────────────────────────────────────
    @PostMapping
    public ResponseEntity<StudentDTO> createStudent(
            @RequestBody StudentCreateRequest request) {  // ← Request DTO com a paràmetre
        StudentDTO createdStudent = studentService.createStudent(request);
        return new ResponseEntity<>(createdStudent, HttpStatus.CREATED);
        // El servei (via Mapper) retorna un StudentDTO — el controlador no el construeix
    }

    // ── GET /api/students/{id} ─────────────────────────────────────────
    @GetMapping("/{id}")
    public ResponseEntity<StudentDTO> getStudentById(
            @PathVariable Long id) {
        StudentDTO student = studentService.getStudentById(id);  // Ja ve convertit
        return ResponseEntity.ok(student);
    }

    // ── GET /api/students ──────────────────────────────────────────────
    @GetMapping
    public ResponseEntity<List<StudentDTO>> getAllStudents() {
        List<StudentDTO> students = studentService.getAllStudents();  // Llista de DTOs
        return ResponseEntity.ok(students);
    }

    // ── DELETE /api/students/{id} ──────────────────────────────────────
    @DeleteMapping("/{id}")
    public ResponseEntity<String> deleteStudent(@PathVariable Long id) {
        studentService.deleteStudent(id);  // El servei valida i elimina
        return ResponseEntity.ok("borrat correctament");
    }
}

Convenció del projecte

2

Crear el Mapper

Crea la classe al paquet mapper/ i anota-la amb @Component perquè Spring la pugui injectar. Implementa els mètodes toDto() i toEntity() amb comprovació de null.

mapper/AbsenceMapper.java — nou fitxer Mapper 
package com.demo.ra4.school.ra4_school.mapper;

import org.springframework.stereotype.Component;
import com.demo.ra4.school.ra4_school.dto.AbsenceDTO;
import com.demo.ra4.school.ra4_school.dto.AbsenceRequest;
import com.demo.ra4.school.ra4_school.model.Absence;

@Component   // ← IMPRESCINDIBLE: registra el bean a Spring
public class AbsenceMapper {

    // ── Entitat JPA → Response DTO ─────────────────────────────────────
    public AbsenceDTO toDto(Absence entity) {
        if (entity == null) return null;  // Null-safe

        AbsenceDTO dto = new AbsenceDTO();
        dto.setId(entity.getId());
        dto.setDate(entity.getDate());
        dto.setJustified(entity.isJustified());
        // Agafem l'ID de l'estudiant sense incloure tot l'objecte Student
        if (entity.getStudent() != null) {
            dto.setStudentId(entity.getStudent().getId());
        }
        return dto;
    }

    // ── Request DTO → Entitat JPA ──────────────────────────────────────
    public Absence toEntity(AbsenceRequest request) {
        if (request == null) return null;  // Null-safe

        Absence entity = new Absence();
        entity.setDate(request.getDate());
        entity.setJustified(request.isJustified());
        // NOTA: no assignem Student aquí — ho farà el Servei
        return entity;
    }
}
3

Adaptar el Servei

Al servei, injecta el nou mapper i crida'l en cada operació: toEntity() abans de guardar i toDto() abans de retornar.

service/AbsenceService.java — nou fitxer Service 
@Service
public class AbsenceService {

    private final AbsenceRepository   absenceRepository;
    private final StudentRepository   studentRepository;
    private final AbsenceMapper       absenceMapper;  // ← injectat per constructor

    public AbsenceService(AbsenceRepository absenceRepository,
                          StudentRepository studentRepository,
                          AbsenceMapper absenceMapper) {
        this.absenceRepository = absenceRepository;
        this.studentRepository = studentRepository;
        this.absenceMapper = absenceMapper;
    }

    // ── Crear falta ────────────────────────────────────────────────────
    public AbsenceDTO create(AbsenceRequest request) {
                Absence entity = absenceMapper.toEntity(request);  // 1. Mapper: Request → Entitat

        // El Mapper no assigna Student — ho fem aquí perquè cal buscar al repo
        Student student = studentRepository.findById(request.getStudentId())
            .orElseThrow(() -> new Error("Estudiant no trobat"));
        entity.setStudent(student);

        Absence saved = absenceRepository.save(entity);        // 2. Repo: guardar
                return absenceMapper.toDto(saved);                     // 3. Mapper: Entitat → DTO
    }

    // ── Consultar per ID ──────────────────────────────────────────────
    public AbsenceDTO findById(Long id) {
        Absence entity = absenceRepository.findById(id)
            .orElseThrow(() -> new Error("Falta no trobada"));
                return absenceMapper.toDto(entity);  // Mapper: Entitat → DTO
    }

    // ── Llistar totes ─────────────────────────────────────────────────
    public List<AbsenceDTO> findAll() {
        List<Absence> entities = absenceRepository.findAll();
        List<AbsenceDTO> dtos = new ArrayList<>();
        for (Absence item : entities) {
                        dtos.add(absenceMapper.toDto(item));  // Mapper en bucle
        }
        return dtos;
    }
}
4

Adaptar el Controlador

El controlador no toca el mapper. Únicament declara els tipus de DTOs als paràmetres i als tipus de retorn.

controller/AbsenceController.java — nou fitxer Controller 
@RestController
@RequestMapping("/api/absences")
public class AbsenceController {

    private final AbsenceService absenceService;

    public AbsenceController(AbsenceService absenceService) {
        this.absenceService = absenceService;

    // ── POST /api/absences ─────────────────────────────────────────────

5

Verificar que tot funciona

Comprova els punts següents per assegurar-te que el Mapper s'integra correctament:

  • @Component al Mapper: sense aquesta anotació, Spring no el registra i la injecció fallaria amb NoSuchBeanDefinitionException.
  • Null-safety: tots els mètodes del Mapper comproven if (input == null) return null; a l'inici.
  • Sense recursivitat: quan un DTO aniuat referencia l'objecte pare, s'usa el constructor sense el camp pare o es deixa el camp a null.
  • El Repositori no veu DTOs: confirma que cap mètode del repositori accepta ni retorna un DTO.
  • El Controlador no crida el Mapper: confirma que el controlador no importa cap classe del paquet mapper/.

Error freqüent

Oblidar @Component al Mapper és l'error més habitual. El compilador no el detecta, però l'aplicació peta en arrencar amb un error d'injecció de dependències.

Resum — responsabilitats de cada capa

Capa Responsabilitat respecte als DTOs i Mappers
DTO (Request) Transporta les dades de la petició HTTP. Cap lògica, cap anotació JPA.
DTO (Response) Transporta les dades de la resposta API. Controla quina informació s'exposa al client.
Mapper (@Component) Únic responsable de convertir entre entitats i DTOs. Té toDto() i toEntity().
Repository Mai veu DTOs. Treballa exclusivament amb entitats JPA. No canvia quan s'afegeixen mappers.
Service Injecta el Mapper. Crida toEntity() abans de guardar i toDto() abans de retornar.
Controller Mai crida el Mapper directament. Rep Request DTOs i retorna Response DTOs gràcies al Servei.
