Conceptes bàsics de seguretat web

Autenticació, autorització, sessions, tokens i JWT

1.1 Autenticació vs Autorització

Dos conceptes que sovint es confonen però que fan coses molt diferents:

ConceptePregunta que responExemple
Autenticació Qui ets? Login amb usuari i contrasenya
Autorització Què pots fer? Només els ADMIN poden veure tots els usuaris

Exemple del món real: Entrar a un edifici amb targeta és autenticació. Que certes portes no s'obrin perquè no tens permís és autorització.

En el nostre projecte, Spring Security gestiona les dues coses: primer comprova qui ets (mitjançant el token JWT) i després comprova si pots accedir a l'endpoint que has demanat (mitjançant els rols USER o ADMIN).

1.2 Sessions vs Tokens (stateful vs stateless)

Quan un usuari fa login, el servidor ha de recordar d'alguna manera que aquell usuari ja s'ha identificat.
Hi ha dues maneres de fer-ho:

Sessions (stateful) — manera clàssica

  • El servidor guarda en memòria que l'usuari X està connectat.
  • El client rep un ID de sessió (normalment en una cookie).
  • A cada petició, el client envia l'ID i el servidor el busca en memòria.
  • Problema: si hi ha molts servidors, tots han de compartir la memòria de sessions.

Tokens (stateless) — manera moderna per a APIs REST

  • El servidor no guarda res.
  • Quan l'usuari fa login, el servidor genera un token i el retorna.
  • El client guarda el token i l'envia a cada petició.
  • El servidor simplement valida el token — no necessita memòria.
  • Ideal per a APIs REST: qualsevol servidor pot validar qualsevol token.

El nostre projecte al ser una APIRest utilitzarem els tokens (stateless) per a l'autenticació mitjançant JWT.

1.3 Què és JWT

JWT (JSON Web Token) és un estàndard per transmetre informació de forma segura entre dues parts, en format text. Conté dades de l'usuari i una firma criptogràfica que garanteix que ningú l'ha modificat.

Un JWT té tres parts separades per un punt (.):

Header

Tipus de token i algorisme de firma.
Ex: {"alg":"HS256","typ":"JWT"}

Payload

Dades de l'usuari (nom, rol, quan expira).
Ex: {"sub":"joan","role":"ADMIN","exp":...}

Signature

Firma digital. Garanteix que el token no ha estat modificat.

Les tres parts es codifiquen en Base64 i s'ajunten amb punts:

eyJhbGciOiJIUzI1NiJ9.eyJzdWIiOiJqb2FuIiwicm9sZSI6IkFETUlOIn0.abc123XYZ

Important: El contingut del JWT és llegible per tothom (Base64 no és xifrat). No hi posis contrasenyes ni dades sensibles. El que garanteix la seguretat és la firma: si algú modifica el payload, la firma deixa de ser vàlida i el servidor el rebutja.

Cicle de vida d'un JWT

  • El token té una data d'expiració (al nostre projecte, 24 hores).
  • Quan expira, l'usuari ha de tornar a fer login per obtenir-ne un de nou.
  • El servidor no pot invalidar un token manualment (és stateless): per això l'expiració curta és important.

1.4 Què és Spring Security

Spring Security és una llibreria de Spring que s'encarrega de protegir la nostra aplicació.
En el moment que l'afegim al projecte:

  • Totes les rutes queden protegides per defecte.
  • Qualsevol petició sense autenticació rep un error 401 Unauthorized.
  • Proporciona un sistema extensible per definir les nostres pròpies regles.

Els tres conceptes clau de Spring Security que usem al projecte:

ConcepteFitxer al projecteFunció
SecurityFilterChain SecurityConfig.java Defineix quines rutes estan protegides i quines no
UserDetailsService ApplicationConfig.java Li diu a Spring Security on trobar els usuaris (BBDD)
UserDetails User.java Representa un usuari amb nom, contrasenya i rols

1.5 Spring Security + JWT

Abans d'entrar al codi, és important entendre el rol de cadascuna de les dues tecnologies que usarem:

TecnologiaRolAnalogia
Spring Security El sistema de seguretat de l'aplicació. Decideix qui pot entrar i a quins llocs pot accedir. El porter d'un edifici.
JWT El format del "carnet d'identitat" digital que rep l'usuari quan fa login. El carnet que el porter comprova a cada visita.

Com treballen junts (conceptualment)

  1. L'usuari prèviament s'ha hagut de registrar
  2. L'usuari fa login amb nom d'usuari i contrasenya.
  3. Si les credencials són correctes, el servidor genera un token JWT i el retorna al client.
  4. A partir d'aquell moment, el client envia el token a cada petició.
  5. Spring Security intercepta les peticions, comprova el token i decideix si deixa passar o no.

Clau: Spring Security és el sistema que protegeix l'aplicació. JWT és el mecanisme que fem servir per identificar l'usuari de manera segura sense sessions. Cadascun té el seu rol i treballen junts.

Important: El registre i el login son endpoints que han de ser públics (no necessiten token) ja que son necessaris per poder obtenir el token.

Spring Security — Coneixements mínims

Com funciona Spring Security amb usuaris a la base de dades

2.1 La cadena de filtres (Filter Chain)

Spring Security funciona com una cadena de filtres. Cada petició HTTP que arriba al servidor passa per una sèrie de filtres abans d'arribar al controlador.

→
Petició HTTP - Arriba a l'APIRest una petició json amb un token
↓
F1
JWT → valida el token (Identifica l'usuari)
↓
F2
(Spring Security) UsernamePasswordAuthenticationFilter → login clàssic que amb JWT no fa res ja que ho autentica el JWT
↓
F3
(Spring Security) AuthorizationFilter → comprova permisos de l'usuari (què pot fer)
↓
→
(Spring APIRest) Controlador rep la petició

2.2 Els usuaris a la base de dades

Per poder autenticar usuaris, Spring Security necessita buscar-los en algun lloc ja que no sap on es guarden (pot ser en BBDD, memòria, LDAP...).
En el cas d'aquest document s'enfoca en l'opció de guardar-ho en una base de dades.

És per això que necessitarem definir una entitat User i haurem d'utilitzar un Repository. És important que tingui el mètode findByUsername() ja que Spring Security l'ha d'utilitzar.

// UserRepository.java
public interface UserRepository extends JpaRepository<User, Long> {
    Optional<User> findByUsername(String username);   // SELECT ... WHERE username = ?
}

Per què és necessari? Spring Security necessita el mètode findByUsername() per buscar l'usuari quan algú intenta fer login. Sense el repository, no es pot connectar la seguretat amb la base de dades.

Com s'utilitza el Repository en cada pas?

1. En el Registre — Crear l'usuari .save()

Aquí el UserRepository serveix per persistir la informació per primera vegada.

  • L'acció: L'usuari envia les dades (username, email, password).
  • El Repository: Fem servir el mètode .save(nouUsuari) per desar-lo a la BBDD.
  • Seguretat: Abans de fer el save, hem de passar el password pel PasswordEncoder (BCrypt) perquè a la BBDD mai hi hagi contrasenyes llegibles.
2. En el Login — Validar l'usuari .findByUsername()

Aquí és on el mètode findByUsername() es converteix en l'estrella.

  • L'acció: Spring Security rep un intent de login amb el nom "joan12".
  • El Repository: Spring (o nosaltres manualment) crida a findByUsername("joan12").
    • Si no existeix: Retorna un error (401 Unauthorized).
    • Si existeix: Recupera l'objecte User sencer (amb la seva contrasenya encriptada i els seus rols).
  • La comparació: Spring compara la contrasenya que ha enviat l'usuari amb la que el Repository ha tret de la BBDD.

2.3 UserDetails i UserDetailsService

UserDetails — qui és l'usuari

És una interfície de Spring Security. El nostre User.java la implementa per dir-li a Spring Security quines dades té l'usuari:

public class User implements UserDetails {
    // ...
    @Override
    public Collection<? extends GrantedAuthority> getAuthorities() {
        return List.of(new SimpleGrantedAuthority("ROLE_" + role.name()));
    }
    @Override
    public String getPassword() { return password; }
    @Override
    public String getUsername() { return username; }
}

UserDetailsService — on buscar l'usuari

Li diem a Spring Security que busqui l'usuari a la BBDD pel nom d'usuari. Si no el troba, llança una excepció:

// ApplicationConfig.java
@Bean
public UserDetailsService userDetailsService() {
    return username -> userRepository.findByUsername(username)
        .orElseThrow(() -> new UsernameNotFoundException("Usuari no trobat"));
}

Important: Spring Security necessita que en el repositori hi hagi el mètode loadUserByUsername() ja que el crida automàticament quan algú intenta fer login o quan el nostre JwtFilter necessita carregar l'usuari a partir del token.

2.4 Contrasenyes: BCrypt

Les contrasenyes mai es guarden en text pla a la base de dades. S'utilitza un algorisme de hash unidireccional anomenat BCrypt:

Contrasenya realEl que es guarda a BBDD
1234$2a$10$abc123xyz456... (60 caràcters)
  • El hash no es pot revertir: no es pot obtenir 1234 a partir del hash.
  • Quan l'usuari fa login, Spring Security fa el hash de la contrasenya introduïda i la compara amb el hash guardat.
  • A ApplicationConfig.java tenim el bean PasswordEncoder que s'usa a tot el projecte.
// ApplicationConfig.java
@Bean
public PasswordEncoder passwordEncoder() {
    return new BCryptPasswordEncoder();
}
// AuthService.java — en crear un usuari nou
.password(passwordEncoder.encode(request.getPassword()))

2.5 Autorització per rols

Un cop l'usuari s'ha identificat, Spring Security pot controlar a quines parts de l'aplicació pot accedir segons el seu rol.

Al nostre projecte hi ha dos rols definits a Role.java:

public enum Role {
    USER,
    ADMIN
}

Hi ha dues maneres de definir l'autorització:

1 — A la configuració general (SecurityConfig)

// SecurityConfig.java
.authorizeHttpRequests(auth -> auth
    .requestMatchers("/api/auth/**").permitAll()   // públic, sense login
    .anyRequest().authenticated()                  // la resta necessita token
)

2 — Directament als controladors (@PreAuthorize)

// UserController.java
@PreAuthorize("hasRole('ADMIN')")
@GetMapping("/admin")
public ResponseEntity<String> adminEndpoint() {
    return ResponseEntity.ok("Només per a administradors");
}

Diferència: La configuració general és per a rutes senceres. @PreAuthorize és per a mètodes concrets dins d'un controlador. Els dos mètodes es poden combinar.

package ra5.demo.jwt.ra5_demo_jwt.model;

import java.util.Collection;
import java.util.List;
import org.springframework.security.core.GrantedAuthority;
import org.springframework.security.core.authority.SimpleGrantedAuthority;
import org.springframework.security.core.userdetails.UserDetails;
import jakarta.persistence.*;

@Entity @Table(name = "users")
public class User implements UserDetails {

    @Id @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;

    @Column(unique = true, nullable = false)
    private String username;

    @Column(unique = true, nullable = false)
    private String email;

    @Column(nullable = false)
    private String password; // sempre guardat com a hash BCrypt

    @Enumerated(EnumType.STRING) // guarda "USER" o "ADMIN", no 0/1
    private Role role;

    public User() {}

    public User(Long id, String username, String email, String password, Role role) {
        this.id = id;
        this.username = username;
        this.email = email;
        this.password = password;
        this.role = role;
    }

    // Getters i setters
    public Long getId() { return id; }
    public void setId(Long id) { this.id = id; }
    public String getEmail() { return email; }
    public void setEmail(String email) { this.email = email; }
    public void setUsername(String username) { this.username = username; }
    public void setPassword(String password) { this.password = password; }
    public Role getRole() { return role; }
    public void setRole(Role role) { this.role = role; }

    // Mètodes obligatoris de UserDetails:
    @Override
    public Collection<? extends GrantedAuthority> getAuthorities() {
        return List.of(new SimpleGrantedAuthority("ROLE_" + role.name()));
    }

    @Override
    public String getPassword() { return password; }

    @Override
    public String getUsername() { return username; }
}

service/JwtService.java Pas 5 @Service

Gestiona tota la lògica relacionada amb el token JWT: generar-lo, validar-lo i extreure'n dades. Llegeix la clau secreta i l'expiració de application.properties.

package ra5.demo.jwt.ra5_demo_jwt.service;

import java.util.Date;
import java.util.function.Function;
import javax.crypto.SecretKey;
import org.springframework.beans.factory.annotation.Value;
import org.springframework.security.core.userdetails.UserDetails;
import org.springframework.stereotype.Service;
import io.jsonwebtoken.Claims;
import io.jsonwebtoken.Jwts;
import io.jsonwebtoken.io.Decoders;
import io.jsonwebtoken.security.Keys;

@Service
public class JwtService {

    @Value("${jwt.secret}") private String secretKey;
    @Value("${jwt.expiration}") private long jwtExpiration;

    /** Extreu el username (subject) del token */
    public String extractUsername(String token) {
        return extractClaim(token, Claims::getSubject);
    }

    /** Genera un token JWT per a l'usuari */
    public String generateToken(UserDetails userDetails) {
        return Jwts.builder()
            .subject(userDetails.getUsername())
            .issuedAt(new Date(System.currentTimeMillis()))
            .expiration(new Date(System.currentTimeMillis() + jwtExpiration))
            .signWith(getSigningKey())
            .compact();
    }

    /** Retorna true si el token és vàlid i no ha caducat */
    public boolean isTokenValid(String token, UserDetails userDetails) {
        String username = extractUsername(token);
        return username.equals(userDetails.getUsername()) && !isTokenExpired(token);
    }

    public <T> T extractClaim(String token, Function<Claims, T> claimsResolver) {
        Claims claims = extractAllClaims(token);
        return claimsResolver.apply(claims);
    }

    private Claims extractAllClaims(String token) {
        return Jwts.parser()
            .verifyWith(getSigningKey())
            .build()
            .parseSignedClaims(token)
            .getPayload();
    }

    private boolean isTokenExpired(String token) {
        return extractClaim(token, Claims::getExpiration).before(new Date());
    }

    private SecretKey getSigningKey() {
        byte[] keyBytes = Decoders.BASE64.decode(secretKey);
        return Keys.hmacShaKeyFor(keyBytes);
    }
}
config/ApplicationConfig.java Pas 6 @Configuration

Proporciona a Spring els beans d'infraestructura que necessita per gestionar l'autenticació: com carregar usuaris, com encriptar contrasenyes i com verificar credencials.

Què és un bean?

Un bean és un objecte que Spring crea, configura i gestiona automàticament. En lloc de fer new MevaClasse() manualment, li diem a Spring que el creï ell amb @Bean i el tingui disponible per injectar allà on faci falta amb @Autowired.

Analogia: Imagina Spring com un magatzem. Els beans són les eines del magatzem. Quan una classe necessita una eina, no la compra ella mateixa — li demana al magatzem i Spring li proporciona la que ja té creada.

Aquest fitxer proporciona els beans d'infraestructura que Spring Security necessita per gestionar l'autenticació:

  • userDetailsService() → li explica a Spring on trobar els usuaris (a MySQL).
  • passwordEncoder() → proporciona BCrypt per encriptar i comparar contrasenyes.
  • authenticationProvider() → combina els dos anteriors per verificar credencials.
package ra5.demo.jwt.ra5_demo_jwt.config;

import org.springframework.beans.factory.annotation.Autowired;
import org.springframework.context.annotation.Bean;
import org.springframework.context.annotation.Configuration;
import org.springframework.security.authentication.AuthenticationProvider;
import org.springframework.security.authentication.dao.DaoAuthenticationProvider;
import org.springframework.security.core.userdetails.UserDetailsService;
import org.springframework.security.core.userdetails.UsernameNotFoundException;
import org.springframework.security.crypto.bcrypt.BCryptPasswordEncoder;
import org.springframework.security.crypto.password.PasswordEncoder;
import ra5.demo.jwt.ra5_demo_jwt.repository.UserRepository;

@Configuration
public class ApplicationConfig {

    @Autowired
    private UserRepository userRepository;

    /** Com carregar un usuari per username (busca a MySQL) */
    @Bean
    public UserDetailsService userDetailsService() {
        return username -> userRepository.findByUsername(username)
            .orElseThrow(() -> new UsernameNotFoundException("Usuari no trobat"));
    }

    /** BCrypt per encriptar contrasenyes */
    @Bean
    public PasswordEncoder passwordEncoder() {
        return new BCryptPasswordEncoder();
    }

    /** Gestiona el procés d'autenticació (carrega user + compara password) */
    @Bean
    public AuthenticationProvider authenticationProvider() {
        DaoAuthenticationProvider provider = new DaoAuthenticationProvider(userDetailsService());
        provider.setPasswordEncoder(passwordEncoder());
        return provider;
    }
}
service/AuthService.java Pas 7 @Service

Conté la lògica de negoci per al registre i el login. Coordina el repositori, el PasswordEncoder, el JwtService i l'AuthenticationManager.

package ra5.demo.jwt.ra5_demo_jwt.service;

import org.springframework.beans.factory.annotation.Autowired;
import org.springframework.security.authentication.AuthenticationManager;
import org.springframework.security.authentication.UsernamePasswordAuthenticationToken;
import org.springframework.security.core.AuthenticationException;
import org.springframework.security.crypto.password.PasswordEncoder;
import org.springframework.stereotype.Service;
import ra5.demo.jwt.ra5_demo_jwt.dto.*;
import ra5.demo.jwt.ra5_demo_jwt.model.Role;
import ra5.demo.jwt.ra5_demo_jwt.model.User;
import ra5.demo.jwt.ra5_demo_jwt.repository.UserRepository;

@Service
public class AuthService {

    @Autowired private UserRepository userRepository;
    @Autowired private PasswordEncoder passwordEncoder;
    @Autowired private JwtService jwtService;
    @Autowired private AuthenticationManager authenticationManager;

    public AuthResponse register(RegisterRequest request) {
        if (userRepository.existsByUsername(request.getUsername())) {
            return new AuthResponse("No es pot registrar l'usuari");
        }

        User user = new User();
        user.setUsername(request.getUsername());
        user.setEmail(request.getEmail());
        user.setPassword(passwordEncoder.encode(request.getPassword()));
        user.setRole(Role.USER);

        userRepository.save(user);
        return new AuthResponse(jwtService.generateToken(user));
    }

    public AuthResponse login(LoginRequest request) {
        try {
            authenticationManager.authenticate(
                new UsernamePasswordAuthenticationToken(
                    request.getUsername(),
                    request.getPassword()
                )
            );
        } catch (AuthenticationException e) {
            return new AuthResponse("Credencials incorrectes");
        }

        User user = userRepository.findByUsername(request.getUsername()).orElseThrow();
        return new AuthResponse(jwtService.generateToken(user));
    }
}
security/JwtFilter.java Pas 8 OncePerRequestFilter

S'executa una vegada per cada petició HTTP. Extreu el token JWT de la capçalera, el valida i si és correcte informa Spring Security de qui és l'usuari. No bloqueja res: qui decideix bloquejar o deixar passar és SecurityConfig.

package ra5.demo.jwt.ra5_demo_jwt.security;

import java.io.IOException;
import org.springframework.beans.factory.annotation.Autowired;
import org.springframework.security.authentication.UsernamePasswordAuthenticationToken;
import org.springframework.security.core.context.SecurityContextHolder;
import org.springframework.security.core.userdetails.UserDetails;
import org.springframework.security.core.userdetails.UserDetailsService;
import org.springframework.stereotype.Component;
import org.springframework.web.filter.OncePerRequestFilter;
import jakarta.servlet.FilterChain;
import jakarta.servlet.ServletException;
import jakarta.servlet.http.HttpServletRequest;
import jakarta.servlet.http.HttpServletResponse;
import ra5.demo.jwt.ra5_demo_jwt.service.JwtService;

@Component
public class JwtFilter extends OncePerRequestFilter {

    @Autowired private JwtService jwtService;
    @Autowired private UserDetailsService userDetailsService;

    @Override
    protected void doFilterInternal(HttpServletRequest req, HttpServletResponse res,
            FilterChain chain) throws IOException, ServletException {

        // 1. Agafar la capçalera Authorization
        String authHeader = req.getHeader("Authorization");

        // 2. Si no hi ha token Bearer, deixem passar sense autenticació
        if (authHeader == null || !authHeader.startsWith("Bearer ")) {
            chain.doFilter(req, res);
            return;
        }

        // 3. Extreure el token (elimina el prefix "Bearer ")
        String jwt = authHeader.substring(7);

        // 4. Extreure el username del token
        String username = jwtService.extractUsername(jwt);

        // 5. Validar i autenticar (si no ho estava ja)
        if (username != null && SecurityContextHolder.getContext().getAuthentication() == null) {
            UserDetails userDetails = userDetailsService.loadUserByUsername(username);

            if (jwtService.isTokenValid(jwt, userDetails)) {
                var authToken = new UsernamePasswordAuthenticationToken(
                    userDetails, null, userDetails.getAuthorities()
                );
                SecurityContextHolder.getContext().setAuthentication(authToken);
            }
        }

        chain.doFilter(req, res);
    }
}
config/SecurityConfig.java Pas 9 @Configuration @EnableMethodSecurity

El fitxer central de seguretat. Defineix quines rutes són públiques, quines necessiten token, desactiva les sessions i afegeix el JwtFilter a la cadena.

package ra5.demo.jwt.ra5_demo_jwt.config;

import org.springframework.beans.factory.annotation.Autowired;
import org.springframework.context.annotation.Bean;
import org.springframework.context.annotation.Configuration;
import org.springframework.security.authentication.AuthenticationManager;
import org.springframework.security.authentication.AuthenticationProvider;
import org.springframework.security.config.annotation.authentication.configuration.AuthenticationConfiguration;
import org.springframework.security.config.annotation.method.configuration.EnableMethodSecurity;
import org.springframework.security.config.annotation.web.builders.HttpSecurity;
import org.springframework.security.config.annotation.web.configurers.AbstractHttpConfigurer;
import org.springframework.security.config.http.SessionCreationPolicy;
import org.springframework.security.web.SecurityFilterChain;
import org.springframework.security.web.authentication.UsernamePasswordAuthenticationFilter;
import ra5.demo.jwt.ra5_demo_jwt.security.JwtFilter;

@EnableMethodSecurity
@Configuration
public class SecurityConfig {

    @Autowired private JwtFilter jwtFilter;
    @Autowired private AuthenticationProvider authenticationProvider;

    @Bean
    public SecurityFilterChain securityFilterChain(HttpSecurity http) throws Exception {
        http
            // Desactiva CSRF: amb JWT no hi ha sessions ni cookies
            .csrf(AbstractHttpConfigurer::disable)
            .authorizeHttpRequests(auth -> auth
                .requestMatchers("/api/auth/**").permitAll()  // login i registre: públics
                .requestMatchers("/swagger-ui/**").permitAll()
                .requestMatchers("/v3/api-docs/**").permitAll()
                .anyRequest().authenticated()                 // la resta necessita token
            )
            // No es creen sessions: cada petició s'identifica amb el token
            .sessionManagement(s -> s
                .sessionCreationPolicy(SessionCreationPolicy.STATELESS)
            )
            .authenticationProvider(authenticationProvider)
            // El nostre JwtFilter s'executa ABANS del filtre estàndard de Spring
            .addFilterBefore(jwtFilter, UsernamePasswordAuthenticationFilter.class);
        return http.build();
    }

    @Bean
    public AuthenticationManager authenticationManager(AuthenticationConfiguration cfg)
            throws Exception {
        return cfg.getAuthenticationManager();
    }
}

3

API REST amb JWT — Com funciona tot el sistema

Des que el client crida un endpoint fins que rep la resposta

3.1 Per què una API REST usa JWT i no sessions?

Una API REST ha de ser stateless (sense estat): cada petició és independent i el servidor no ha de recordar res entre peticions. Les sessions van en contra d'aquest principi perquè obliguen el servidor a guardar informació.

3.2 Flux complet: registre d'un usuari nou

1
Client fa POST /api/auth/register amb { username, email, password }
↓
2
AuthController rep la petició i crida a AuthService.register()
↓
3
AuthService comprova si el nom d'usuari ja existeix a BBDD
↓
4
Crea el nou User amb la contrasenya feta hash amb BCrypt i rol USER
↓
5
Desa l'usuari a la BBDD via UserRepository
↓
6
JwtService genera un token JWT amb el nom d'usuari i l'expiració (24h)
↓
7
El servidor retorna { "token": "xxxxx.yyyyy.zzzzz" } amb codi 201 Created

3.3 Flux complet: login

1
Client fa POST /api/auth/login amb { username, password }
↓
2
AuthController crida a AuthService.login()
↓
3
AuthenticationManager verifica les credencials: carrega l'usuari de BBDD, fa hash de la contrasenya i compara
↓
4
Si les credencials són incorrectes → excepció → resposta 401 Unauthorized
↓
5
Si són correctes → JwtService genera un nou token JWT
↓
6
Retorna { "token": "xxxxx.yyyyy.zzzzz" } amb codi 200 OK

3.4 Flux complet: petició a un endpoint protegit

Un cop el client té el token, l'envia a la capçalera HTTP de cada petició:

GET /me HTTP/1.1
Authorization: Bearer eyJhbGciOiJIUzI1NiJ9.eyJzdWIiOiJqb2FuIn0.abc123
1
La petició arriba al servidor amb la capçalera Authorization: Bearer <token>
↓
2
JwtFilter intercepta la petició i extreu el token de la capçalera
↓
3
JwtService extreu el nom d'usuari (subject) del token
↓
4
Carrega l'usuari de la BBDD via UserDetailsService
↓
5
Valida el token: ¿el nom d'usuari coincideix? ¿ha expirat?
↓
6
Si és vàlid → posa l'autenticació al SecurityContext (Spring Security ja sap qui és)
↓
7
SecurityConfig comprova si l'usuari té el rol necessari per accedir a l'endpoint
↓
8
Si té permís → el controlador processa la petició i retorna la resposta

Que passa si el token no és vàlid? El JwtFilter no posa res al SecurityContext. Spring Security veu que la petició no està autenticada i retorna 401 Unauthorized automàticament, sense arribar al controlador.

3.5 Generació del token JWT (JwtService)

El JwtService és la classe encarregada de crear i validar els tokens. Treballa amb la llibreria JJWT.

Generació del token

public String generateToken(UserDetails userDetails) {
    return Jwts.builder()
        .subject(userDetails.getUsername())   // qui és l'usuari
        .issuedAt(new Date())                 // quan s'ha creat
        .expiration(new Date(System.currentTimeMillis() + jwtExpiration)) // quan expira
        .signWith(getSigningKey())            // firma amb la clau secreta
        .compact();                           // retorna el token com a String
}

Validació del token

public boolean isTokenValid(String token, UserDetails userDetails) {
    final String username = extractUsername(token);
    return (username.equals(userDetails.getUsername())) && !isTokenExpired(token);
}
